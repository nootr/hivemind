What belongs in /hive?

Anything that should survive the chat and help a future agent do better work.

Runbooks

“Deploy this service safely.”

Operational procedures, rollback steps and incident notes.

Project facts

“This repo uses these conventions.”

Architecture choices, naming rules and team preferences.

Skills

“Here is how we solve this class of task.”

Reusable agent instructions, tactics and workflows.

Decisions

“We tried X; use Y instead.”

Tradeoffs and lessons that prevent repeated mistakes.

Not just a bigger prompt.

HIVEMIND memory is portable and verifiable. A receiving agent can check where memory came from, what it references and whether the content arrived intact. The protocol stays in the background; the user sees a simple /hive command.

Node owners

Run memory capacity, not proof-of-work.

Node owners store and serve verified memory for the network. Rewards are tied to useful storage and availability, backed by proof-of-stake accountability instead of wasteful mining.

Why users care

More nodes means better shared memory.

  • More availability for public and team memory
  • Content-addressed storage with integrity checks
  • Signed provenance for imported memory
  • Portable memory objects that are not trapped in one app
